# Development Story

UnderWatch 개발 중 발생한 주요 문제와 해결 과정을 정리한 문서입니다.

---

## 1. Spring 6 전환 문제

### 문제

Spring 4~5 기준 예제를 그대로 적용했을 때 실행 오류가 발생했습니다.

### 원인과 해결

| 문제                              | 원인                            | 해결                                    |
| ------------------------------- | ----------------------------- | ------------------------------------- |
| 파일 업로드 오류                       | `CommonsMultipartResolver` 제거 | `StandardServletMultipartResolver` 적용 |
| `@RequestParam` 이름 인식 실패        | 파라미터명 미보존                     | `-parameters` 옵션 추가                   |
| `@AuthenticationPrincipal` null | 인자 리졸버 미등록                    | ArgumentResolver 등록                   |
| JSP Security 태그 오류              | 표현식 핸들러 미노출                   | ExpressionHandler 빈 등록                |

### 배운 점

버전이 바뀌면 예제 코드가 그대로 동작하지 않을 수 있으므로 공식 문서와 실제 버전을 함께 확인해야 합니다.

## 2. Oracle 한글 인코딩 문제

### 문제

SQL*Plus로 스크립트를 실행하면 한글 데이터가 깨졌습니다.

### 원인

* SQL*Plus 클라이언트 인코딩 문제
* DB 캐릭터셋 불일치
* CDB / PDB 접속 혼동
* 시퀀스 재실행 충돌

### 해결

* SQL은 ASCII 중심으로 작성
* 한글 데이터는 `UNISTR` 사용
* 접속 DB 확인
* 시퀀스 보정 스크립트 작성

### 배운 점

DB 문제는 SQL 문법뿐 아니라 인코딩, 접속 대상, 시퀀스 상태까지 함께 확인해야 합니다.

## 3. Eclipse / Tomcat 배포 문제

### 문제

코드를 수정했는데 화면에 반영되지 않거나 서버 실행 오류가 발생했습니다.

### 해결

* 서버 모듈 재등록
* 컨텍스트 루트 `/`로 수정
* Clean 후 재시작
* 브라우저 강력 새로고침
* 중복 서버 인스턴스 정리

### 배운 점

웹 프로젝트에서는 코드 오류와 배포 오류를 구분하는 습관이 중요합니다.

## 4. 인증 확장 문제

### 문제

폼 로그인에서 소셜 로그인, 얼굴 로그인으로 확장하면서 기존 코드가 깨질 가능성이 있었습니다.

### 해결

`OwUserPrincipal`을 만들어 세 가지 로그인 방식이 모두 같은 Principal을 사용하도록 통합했습니다.

```text
Form Login  ─┐
OAuth Login ─┼─▶ OwUserPrincipal
Face Login  ─┘
```

### 배운 점

기능을 추가할 때 기존 Controller와 JSP가 인증 방식에 의존하지 않도록 공통 구조를 먼저 설계하는 것이 중요합니다.

## 7. 커뮤니티 운영 기능 보강

### 비회원 게시글

회원 로그인 없이도 글을 쓸 수 있도록 `ow_board.writer_id`를 NULL 허용으로 바꾸고,
`guest_name`, `guest_password` 컬럼을 추가했습니다. 비밀번호는 평문이 아니라 BCrypt 해시로 저장하고,
수정/삭제 요청 때마다 다시 검증합니다.

### 댓글 추천/비추천

게시글 추천 구조와 같은 원칙으로 `ow_comment_vote(member_id, comment_no)` 복합 PK를 두었습니다.
한 회원이 같은 댓글에 여러 번 투표하지 못하게 DB 제약과 서비스 로직을 함께 사용했습니다.

### 탈퇴 7일 유예

즉시 삭제 대신 `WITHDRAWN` 상태와 `withdraw_at`을 기록합니다. 7일 안에 다시 로그인하면
`LoginSuccessHandler`가 계정을 복구하고, 7일이 지난 계정은 매일 새벽 4시 `WithdrawScheduler`가 삭제합니다.

### 실시간 배지

쪽지/알림을 생성한 뒤 DB의 안 읽은 개수를 다시 조회하고 `NotifyWebSocketHandler`로 브라우저에 전송합니다.
받는 사용자가 여러 탭을 열어도 사용자 ID별 세션 목록에 모두 전송하도록 구성했습니다.

## 8. 얼굴 인식 로그인의 시행착오

### 문제

"얼굴로 로그인" 기능을 어떻게 만들지를 두고 네 번이나 방식을 갈아엎었습니다.

### 과정 (WebAuthn → face-api.js → LBPH/PCA 실험 → Face API)

1. **WebAuthn** — 처음엔 표준 생체 인증(WebAuthn/FIDO2)으로 시도했습니다. 그러나 이는 **기기의 인증기(Windows Hello 등)에 위임**하는 방식이라, "임의의 웹캠으로 얼굴을 인식해 누구인지 찾는" 기능이 아니었습니다. 기기·HTTPS에 묶여 커뮤니티 웹앱과 맞지 않아 포기했습니다.
2. **face-api.js** — 실제 웹캠 얼굴 인식을 위해 브라우저 딥러닝 라이브러리를 도입했습니다. 잘 동작했지만 **사전학습 모델을 CDN에서 가져다 쓰는** 방식이었습니다.
3. **LBPH/PCA 실험** — 외부 모델 없이 Java에서 직접 비교하는 방식도 구현했습니다. 다만 조명·각도 변화에 약했고, 실사용 안정성이 부족했습니다.
4. **Face API 연동** — 최종 흐름은 Python FastAPI 얼굴 서버에서 InsightFace/ArcFace 임베딩을 받고, Spring에서 DB에 저장한 최대 3개 임베딩과 코사인 유사도로 1:N 비교하는 구조로 정리했습니다.

### 배운 점

* "얼굴 인증"이라고 다 같은 것이 아니다 — **기기 인증(WebAuthn)** 과 **앱이 자체적으로 식별하는 1:N 얼굴 인식**은 전혀 다른 문제였습니다.
* "가져다 쓰는 것"과 "서비스에서 책임지는 것"의 경계를 고민하게 됐고, 얼굴 검출/임베딩은 전용 API로, 회원 식별/임계값/세션 처리는 Spring에서 책임지는 구조로 정리했습니다.
